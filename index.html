GIT - podstawy

Co to jest?

Jest to kontener zawierające dane na temat zmian, jakie zaszły w naszym projekcie (kto je wprowadził, kiedy, czego się tyczyły) oraz także informacje o teraźniejszym stanie naszego projektu, które możemy w każdej chwili pobrać na swoją lokalną maszynę. Git umożliwia nam także cofanie się do zmian oraz przywracanie ich. Dane o tworzonym przez nas projekcie przechowywane są, w tak zwanym repozytorium.
Repozytorium możemy podzielić na 2 części:

  • Repozytorium lokalne (local) → to, które znajduje się na naszym komputerze
  • Repozytorium zdalne (remote) → jest to repozytorium znajdujące się na serwerach, chmurach lub serwisach takich jak GitHub, GitLab czy BitBucket

Instalacja i konfiguracja

Instalacja

Aby zainstalować git na naszym komputerze, to w zależności od systemu operacyjnego, powinniśmy wykonać odpowiednie kroki

Linux

  1. Otwórz terminal
  2. Wpisz następującą komendę sudo apt-install git. Powinna ona zainstalować git na Twoim komputerze
  3. Wpisz następującą komendę git --version. Jeżeli na ekranie pojawił się napis z wersją gita proces instalacji przebiegł pomyślnie

Windows

  1. Pobierz git z tej strony Downloading Git
  2. Zainstaluj git, korzystając z pobranego pliku (wybierz ustawienia rekomendowane)
  3. W wyszukiwarce Windows wyszukaj Git bash i odpal go
  4. W odpalonej konsoli wpisz polecenie git --version i sprawdź, czy zwróciła wersje git jeżeli tak to proces instalacji przebiegł pomyślnie

Konfiguracja

Po instalacji git należy go skonfigurować, zanim zacznie się z niego korzystać. Jednymi z opcj, które można skonfigurować to: nazwy użytkownika, email, edytora tekstu, kolorowania składni itd. . Najważniejsze są jednak ustawienia username oraz emial ponieważ bez nich nie można przekazać zmiany do zdalnego repozytorium. Aby skonfigurować, ustawienia globalne należy dodać flagę --gloabl a jeżeli konfiguracja ma dotyczyć tylko jednego konkretnego repozytorium, to nie dodajemy flagi.
Globalna konfiguracja email i username:

git config --global usr.name [your.sername]
git config --global user.email [your.email]

Jeżeli korzystasz z windowsa powyższe komendy wpisz w aplikajci Git bash, która powinna znajdować się na Twoim komputerze. Od tego mementu zamiast pisać Git bash będę uzywał sformułowania terminal

Aby sprawdzić swoją konfigurację, wpisz do terminala polecenie git config --list. Jeżeli bezbłędnie przepisałeś powyższe komendy powinieś zobaczyć takie informacje:

user.name=yourUserName
user.email=your.email@domain.comm

Pierwsze repozytorium

Aby powołać git do życia w naszym projekcie, należy wpisać w terminalu znajdując się w katalogu, który chcemy śledzić polecenie:

git init

Polecenie to wygeneruje plik .git oraz zacznie śledzić projekt i zapisywać informacje o kolejnych zmianach, które zaszły plikach od ostatniego commita (zapisanej zmiany).
Aby sprawdzić informacje na temat projektu oraz plików, które zawierają się w repozytorium, należy wpisać polecenie:

git status

Komenda ta będzie informowała o statusie zmian, które zaszły w konkretnych plikach
Od razu po incjalizacji git po wpisaniu komendy git status w terminalu powinno pojawić się:

On branch master
No commits yet
nothing to commit (create/copy files and use "git add" to track)

W takim razie dodajmy nowy plik do naszego katalogu i wpiszemy komendę git status w terminalu. Komenda powinna wyświetlić informację, że plik, który dodaliśmy jest w stanie untrack i jest zaznaczony na czerwony kolor:

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	test.txt

Następnie wpiszemy komendę:

  • git add file/name/to.add -> dodaje jeden konkretny plik
  • git add . -> dodaje wszystkie pliki, które są śledzone i wykryto w nich zmiany lub są nowymi plikami

Ponowne wpisanie komendy git status powinno podświetlić nowo dodany plik na kolor zielony i przenieść go do sekcji Changes to be committed:

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   test.txt

Następnie można zmodyfikować dodany plik i kolejny raz wpisać komendę git status. Komenda powinna wyświetlić informację, że plik znajduje się w 2 sekcjach: Changes to be committed, Changes not staged for commit.

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   test.txt

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   test.txt

W sekcji not staged for commit, plik ma stan modyfied co mówi o tym, że jest on zmodyfikowany. Aby uwzględnić wprowadzone w nim zmiany, w zapisie stanu, należy go ponownie dodać przy pomocy komendy git add file.name. Po wpisaniu tej komendy i ponownym wpisaniu git status powinniśmy widzieć tylko jedną sekcję z nowym zmodyfikowanym plikiem:

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   test.txt

Teraz usuńmy zmodyfikowany plik i ponownie wpiszmy komendę git status. Komenda powinna wyświetlić informację, że plik znajduje się w 2 sekcjach: Changes to be committed, Changes not staged for commit.

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   test.txt

Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	deleted:    test.txt

W sekcji not staged for commit, plik ma stan deleted co mówi o tym, że został on usunięty. Aby uwzględnić to, że został usunięty, w zapisie stanu, należy go ponownie dodać przy pomocy komendy git add file.name. Po wpisaniu tej komendy i ponownym wpisaniu git status nie powinniśmy zobaczyć żadnego pliku, ponieważ od inicjalizacji nic się nie zmieniło (dodaliśmy plik, który potem usnęliśmy, więc koniec końców nie zaszła żadna zmiana):

On branch master
No commits yet
nothing to commit (create/copy files and use "git add" to track)

Statusy

  • untracked → jest to status, który mówi o tym, że git zauważył zmianę w katalogu, ale nie śledzi tego pliku i nie obserwuje zmian strukturalnych wewnątrz niego
  • new file → status ten informuje nas o tym, że plik jest nowy i dopiero co został dodany do repozytorium
  • modified → status ten mówi nam o tym, że w pliku, który jest śledzony zaszły zmiany
  • deleted → jest to status, który mówi o tym, że plik ten był wcześniej w repozytorium, ale został usnięty
  • renamed → status ten określa plik, który wcześniej istniał w repozytorium, lecz zmieniono mu nazwę

Zatwierdzanie zmian

Aby zatwierdzić zmiany w repozytorium, dla plików znajdujących się w sekcji Changes to be committed można posłużyć się następującymi komendami:

  • git commit → zatwierdza zmiany oraz wyświetla okno, w którym można dodać informacje na temat commita (nazwę, opis)
  • git commit -m "commit changes description" → zatwierdza zmiany w śledzonych plikach i automatycznie dodaje nazwę do zatwierdzonego stanu

Zdalne repozytorium i przekazywanie informacji

Zdalne repozytorium

Jeżeli chcemy przechowywać informacje dotyczące projektu nie tylko u nas na komputerze, ale też gdzieś w sieci tak, aby można udostępnić projekt innym osobom, to możemy założyć konto na platformach do tego przeznaczonych takich jak GitHub lub GitLab itp…
Aby połączyć lokalne repozytorium ze zdalnym, należy skorzystać z komendy:

  • git remote add git@cloud.name:username/repo-name → komenda ta ustawi miejsce, na które będziemy przekazywać zmiany z lokalnego repozytorium

Przekazywanie informacji

  • git push → komenda ta wypchnie zmiany, które wprowadziliśmy na lokalny repozytorium na repozytorium zdalne. Do komendy push możemy dodać flagi takie jak:
    • u origin branchName → jest to flaga, z której korzysta się za pierwszym razem, aby wskazać miejsce na zdalnego repozytorium, na które mają trafiać informacje z repozytorium lokalnego
  • git fetch → aktualizuje informacje na temat zdalnego repozytorium
  • git pull → pobiera informacje ze zdalnego repozytorium i aktualizuje o nie informacje w lokalnym repozytorium
  • git clone https://cloud.name/repo-name → pobiera zdalne repozytorium na nasz komputer
